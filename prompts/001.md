[アプリケーションの概要]

- タイトル:
    - 高速道路RTAアプリ

- UI:
    - Wake Lock APIによるスリープ防止
        - 「運転中の操作禁止」などの警告画面（初回のみ、または毎回）を表示するステップを挟むことで、ロックを取得するためにユーザー操作を発生させる
    - ダークモード切り替え
    - 地図: ベースマップは地理院タイル
    - 測位結果リスト: チェックポイントごとのラップタイムのみを表示
    - リアルタイム表示: 走行中、現在の時速や次のCPまでの距離などをリアルタイムに表示する機能
        - 走行中に最も大きく表示すべきなのは「次のCPまでの距離（現在地からCP座標への直線距離）」
        - 時速表示には、Geolocation APIが返す coords.speed（端末の計算値）を使用
    - 記録の「計測開始」・「計測終了」ボタン
    - 複数の「挑戦記録」をCSV形式でエクスポートする機能
    - GeoJSON形式での走行経路エクスポート
    - 設定画面
        - 測位頻度の秒数・メートル数を設定する機能
        - 古い挑戦記録を削除する機能（自動削除は不要）
    - PWA化（マニフェストファイル等の作成）
        - 地図のキャッシュ: PWAとして、地図タイルを事前にキャッシュ（オフライン利用）する

※バックグラウンド動作はせず、常時フォアグラウンドに表示される想定

- 機能:
    - GPSによる測位
        - 5秒ごと、または移動距離が5m以上の時
        - 一時停止機能: トイレ休憩などでPAに長時間滞在する場合、計測を「一時停止（位置だけは追い続けるが記録はしない）」する
    - 測位結果の経緯度と時刻をIndexedDBに記録
    - 走行経路の可視化（測位結果を直線で結ぶ）
        - 明らかにエラー（外れ値）と考えられる測位結果はプロットしない
    - 挑戦記録の保存
        -「挑戦記録」の単位: 1回の「計測開始から終了」を1つの「セッション」として管理
        - セッション数や測位結果数に制限は設けない
    - JSON配列として定義した各チェックポイントに最も近づいた測位結果をもとに、チェックポイント間の走行に要した時間を算出
        - 走行中にリアルタイムで「今、〇〇を通過！」と判定させ、バイブレーションと音声読み上げ（Web Speech API）（例：「三芳PAを通過しました」）でユーザーに知らせる
    - 経緯度から距離を算出する際には、計算精度を担保するためにライブラリ（geographiclibのJS版など）を使用
        - ヒュベニの公式より厳密かつ根拠があること

[チェックポイント（CP）の管理]

- ルートの選択:
    - 現在の定義は関越道（川越～所沢）の一部だが、将来的に「東北道ルート」「東名道ルート」など、複数のルート（CPセット）をアプリ上で切り替えて選択できるようにUIを構築する
- CP定義の持たせ方:
    - CPのデータ（JSON）はアプリにハードコードする（主催者が定め、利用者は変更できないようにするため）
    - 途中のPAを飛ばしても、最後にゴール（終点IC）に着けば「完走」とみなす

[通過判定ロジック]

- 判定の閾値（半径）: CPの座標から半径50m以内に入った後の最短距離地点を通過時刻とする
- 通過の確定タイミング: CPに最も接近して、再び離れ始めた瞬間に「通過」と確定させる（最も正確に、わずかに遅延することは許容する）
    - 移動平均の窓数: マップ表示用の移動平均は3点
- 通過順序の制約: CPは定義された配列の順番通りに通過する必要がある
- 逆走は考慮しなくてよいが、途中のPAを飛ばすことも可能
    - ただし、飛ばした場合でも本線走行中に通過判定は行う
    - 「最短距離」の厳密性: CPに接近して離れた瞬間に「通過」と判定する際、もし「離れた」と思った直後にさらに近い地点が計測された場合（GPSの揺らぎ）には、記録を上書き更新する（半径1km以内だけ）
        - 半径1kmを過ぎたら、上書き更新せずに、次のCPを追い始める
        - 通知タイミングは1回のみ（更新しない）

[測位結果の外れ値判定ロジック]

- 前回の測位地点からの移動速度が時速180kmを超えている場合
- Geolocation APIが返す accuracy（精度）の値が100m以上を示している場合
- 移動平均
    - マップに表示する際の経緯度のみ移動平均で外れ値判定を行い、IndexedDBに記録したり時速表示すしたりする際は生のデータをそのまま使用する
- トンネル内・電波喪失時の挙動: 長いトンネル等でGPSが途切れた場合、トンネル内のCPをスキップしたとみなして、復帰後の位置から逆算して通過判定を行う
    - トンネルを抜けてGPSが復帰した際、未通過のCPがトンネル内にあった場合、その通過時刻の算出には、GPSが途切れた地点と復帰した地点の時間を等間隔で割る

```json

    {
        name: "川越IC",
        latitude: 35.893205,
        longitude: 139.459547
    },
    {
        name: "三芳PA",
        latitude: 35.844391,
        longitude: 139.503090
    },
    {
        name: "所沢IC",
        latitude: 35.809777,
        longitude: 139.529614
    }

```

[エクスポート仕様]

- CSVの項目: 日時、CP名、ラップタイム、スプリットタイムの他に、全ログの座標データ（経緯度・時刻・速度）も含める
    - 1つのCSVファイルに、全セッションの、CP通過行と生ログ行が混在する形式（フラグ列で区別する）
    - 並びは時刻順
- GeoJSON: 1セッション＝1つのLineString（またはMultiLineString）として出力する

- フロントエンド: HTML, CSS, JavaScript
- 地図ライブラリ: MapLibre GL JS
