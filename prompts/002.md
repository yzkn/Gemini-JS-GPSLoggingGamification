# 技術要件定義書：高速道路RTAアプリ

## 1. アプリケーション概要
*   **名称:** 高速道路RTAアプリ
*   **コンセプト:** 高速道路のチェックポイント（CP）間の通過タイムを競う、ドライバー向け計測Webアプリケーション。
*   **ターゲット:** スマートフォン（Webブラウザ/PWA）
*   **主要動作環境:** フォアグラウンドでの常時表示を前提とする（バックグラウンド動作は考慮しない）。

## 2. 技術スタック
*   **Frontend:** Vanilla JavaScript (ES6+), HTML5, CSS3
*   **Framework:** Vite (ビルドツール)
*   **CSS:** TailwindCSS (PostCSS経由)
*   **Map:** MapLibre GL JS (ベースマップ：国土地理院ベクトル/ラスタタイル)
*   **Calculation:** geographiclib (高精度な距離計算用)
*   **Storage:** IndexedDB (セッションデータ、ログ保存用)
*   **Deployment:** Cloudflare Pages
*   **API:** Geolocation API, Wake Lock API, Web Speech API, Service Worker (PWA)

## 3. 機能要件

### 3.1 測位・記録ロジック
*   **測位頻度:** 5秒間隔 または 5m以上の移動ごとに実行（設定画面で変更可能）。
*   **外れ値除去（フィルタリング）:**
    *   `coords.accuracy` > 100m のデータは破棄。
    *   前回測位地点からの計算速度が 180km/h を超えるデータは破棄。
*   **平滑化:** 地図表示用のプロットのみ、直近3点の移動平均を使用。記録用（IndexedDB）および時速表示には生のGPSデータを使用。
*   **一時停止機能:** 計測セッション中、位置追跡は継続するが、走行記録（ログ保存およびCP判定）を一時的に停止する。

### 3.2 チェックポイント（CP）通過判定アルゴリズム
*   **基本判定:** CP座標から半径50m以内に入り、その中で「最もCPに近づいた（最短距離を記録した）瞬間」を通過時刻とする。
*   **確定タイミング:** 最短距離を記録した後、距離が離れ始めた瞬間に「通過」と確定させる。
*   **補正処理:** 「離れた」と判定した直後でも、半径1km以内であれば、より短い距離が観測された場合に通過時刻・地点を上書き更新する。1kmを超えたら次のCP判定へ移行する。
*   **順序制約:** CPは定義された配列順に通過する必要がある。
*   **トンネル/ロスト補完:** 長いトンネル等でGPSが途切れた際、復帰地点までに未通過のCPがあった場合、ロスト直前地点と復帰地点を直線で結んだものと仮定し、時間を距離で按分（線形補間）して通過時刻を算出する。

### 3.3 ユーザーインターフェース (UI)
*   **スタート画面:**
    *   利用規約/安全上の警告を表示。
    *   ユーザー操作（「同意して開始」ボタン等）をトリガーに **Wake Lock API** を有効化し、スリープを防止。
*   **メイン画面:**
    *   **地図:** MapLibre GL JSによる現在地と走行経路の表示。
    *   **リアルタイム情報:**
        *   「次のCPまでの直線距離」を最優先で大きく表示。
        *   現在の時速（`coords.speed`）を表示。
    *   **ラップリスト:** 通過済みCPのラップタイムを表示。
    *   **コントロール:** 「計測開始」「計測終了」「一時停止」ボタン。
*   **設定画面:**
    *   ルート選択（関越道ルート、東北道ルート等。データはハードコードされたJSONより選択）。
    *   測位頻度設定（秒数、メートル）。
    *   履歴削除機能。
    *   ダークモード切り替え。

### 3.4 音声・通知
*   CP通過確定時、**Web Speech API** を用いて「[CP名]を通過しました」と読み上げ、同時にバイブレーションを実行。

### 3.5 PWA / オフライン対応
*   マニフェストファイルの作成。
*   Service Workerによる地理院タイルの事前キャッシュ機能の実装。

## 4. データ管理仕様

### 4.1 CP定義データ構造 (JSON)
```json
[
  {
    "id": "kan-etsu-down",
    "routeName": "関越川越（下り）",
    "checkpoints": [
      { "name": "川越IC", "latitude": 35.893205, "longitude": 139.459547 },
      { "name": "三芳PA", "latitude": 35.844391, "longitude": 139.503090 },
      { "name": "所沢IC", "latitude": 35.809777, "longitude": 139.529614 }
    ]
  },
  {
    "id": "kan-etsu-up",
    "routeName": "関越川越（上り）",
    "checkpoints": [
      { "name": "所沢IC", "latitude": 35.807816, "longitude": 139.530606 },
      { "name": "三芳PA", "latitude": 35.839042, "longitude": 139.503693 },
      { "name": "川越IC", "latitude": 35.892501, "longitude": 139.459501 }
    ]
  }
]
```

### 4.2 エクスポート仕様
*   **CSV:**
    *   構成: 1ファイルに全セッションのログとCP通過情報を混在させる。
    *   カラム: `timestamp`, `type` (LOG/CP), `cp_name`, `latitude`, `longitude`, `speed`, `lap_time`, `split_time`
    *   ソート: 時系列順。
*   **GeoJSON:**
    *   1セッションを1つの `LineString` として出力。

## 5. 開発・ビルド手順（Build.md 相当）

### プロジェクトセットアップ
```bash
# プロジェクト作成
npm create vite@latest highway-rta-app -- --template vanilla
cd highway-rta-app

# 依存ライブラリのインストール
npm install tailwindcss postcss autoprefixer
npm install maplibre-gl geographiclib
npx tailwindcss init -p
```

### ディレクトリ構成案
- `index.html`
- `src/`
    - `main.js` (エントリーポイント)
    - `map.js` (MapLibre関連)
    - `gps.js` (Geolocation, 通過判定ロジック)
    - `storage.js` (IndexedDB関連)
    - `speech.js` (Web Speech API)
    - `utils.js` (geographiclibを使用した計算等)
- `public/`
    - `manifest.json`
    - `sw.js` (Service Worker)

## 6. 実装上の注意
1.  **高精度距離計算:** 2点間の距離算出には `geographiclib` の測地線計算（Vincenty法と同等かそれ以上の精度）を使用し、ヒュベニの公式は使用しないこと。
2.  **バッテリー消費:** Wake LockとGPSを併用するためバッテリー消費が激しい。UI上で注意喚起することが望ましい。
3.  **セキュリティ:** Cloudflare Pagesでデプロイするため、HTTPSが必須となる（Geolocation APIの要件）。

- 機能:
    - GPSによる測位
        - 5秒ごと、または移動距離が5m以上の時
        - 一時停止機能: トイレ休憩などでPAに長時間滞在する場合、計測を「一時停止（位置だけは追い続けるが記録はしない）」する
    - 測位結果の経緯度と時刻をIndexedDBに記録
    - 走行経路の可視化（測位結果を直線で結ぶ）

    - 経緯度から距離を算出する際には、計算精度を担保するためにライブラリ（geographiclibのJS版など）を使用
        - ヒュベニの公式より厳密かつ根拠があること

[チェックポイント（CP）の管理]

- ルートの選択:
    - 現在の定義は関越道（川越～所沢）の一部だが、将来的に「東北道ルート」「東名道ルート」など、複数のルート（CPセット）をアプリ上で切り替えて選択できるようにUIを構築する
- CP定義の持たせ方:
    - CPのデータ（JSON）はアプリにハードコードする（主催者が定め、利用者は変更できないようにするため）
    - 途中のPAを飛ばしても、最後にゴール（終点IC）に着けば「完走」とみなす

[通過判定ロジック]

- 判定の閾値（半径）: CPの座標から半径50m以内に入った後の最短距離地点を通過時刻とする
- 通過の確定タイミング: CPに最も接近して、再び離れ始めた瞬間に「通過」と確定させる（最も正確に、わずかに遅延することは許容する）
    - 移動平均の窓数: マップ表示用の移動平均は3点
- 通過順序の制約: CPは定義された配列の順番通りに通過する必要がある
- 逆走は考慮しなくてよいが、途中のPAを飛ばすことも可能
    - ただし、飛ばした場合でも本線走行中に通過判定は行う
    - 「最短距離」の厳密性: CPに接近して離れた瞬間に「通過」と判定する際、もし「離れた」と思った直後にさらに近い地点が計測された場合（GPSの揺らぎ）には、記録を上書き更新する（半径1km以内だけ）
        - 半径1kmを過ぎたら、上書き更新せずに、次のCPを追い始める
        - 通知タイミングは1回のみ（更新しない）

[測位結果の外れ値判定ロジック]

- 前回の測位地点からの移動速度が時速180kmを超えている場合
- Geolocation APIが返す accuracy（精度）の値が100m以上を示している場合
- 移動平均
    - マップに表示する際の経緯度のみ移動平均で外れ値判定を行い、IndexedDBに記録したり時速表示すしたりする際は生のデータをそのまま使用する
- トンネル内・電波喪失時の挙動: 長いトンネル等でGPSが途切れた場合、トンネル内のCPをスキップしたとみなして、復帰後の位置から逆算して通過判定を行う
    - トンネルを抜けてGPSが復帰した際、未通過のCPがトンネル内にあった場合、その通過時刻の算出には、GPSが途切れた地点と復帰した地点の時間を等間隔で割る
